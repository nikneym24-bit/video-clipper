# Процесс разработки: Orchestrator-Worker v3

> **Версия:** 1.0
> **Последнее обновление:** 2026-03-03

Нормативный документ. Описывает гибридный паттерн разработки: Opus пишет зависимый код напрямую, Sonnet-агенты получают изолированные модули параллельно.

---

## Архитектура

```
┌──────────────────────────────────────────────────────────────┐
│                     Opus (Оркестратор)                        │
│                                                              │
│  План · Фундамент · Зависимый код · Интеграция · Верификация │
│                                                              │
│              ┌──────────┴──────────┐                         │
│              ▼                     ▼                         │
│       ┌─────────────┐     ┌─────────────┐                    │
│       │   Sonnet A  │     │   Sonnet B  │  ...               │
│       │  (воркер)   │     │  (воркер)   │                    │
│       │ selector.py │     │ publisher.py│                    │
│       └──────┬──────┘     └──────┬──────┘                    │
│              └──────────┬──────────┘                         │
│                         ▼                                    │
│              Opus: интеграция + pytest                        │
└──────────────────────────────────────────────────────────────┘
```

**Opus** — единственная точка управления. Пишет зависимый код, делегирует изолированные модули.
**Sonnet-агенты** — запускаются через `Agent tool` с `model="sonnet"`, `subagent_type="general-purpose"`.

---

## Роли

| Роль | Модель | Что делает | Что НЕ делает |
|------|--------|------------|---------------|
| **Оркестратор** | Opus | План, архитектура, зависимый код, glue code, `__main__.py`, интеграция, верификация | НЕ делегирует GPU-код и security-critical код |
| **Воркер** | Sonnet | Изолированный модуль по спеке (бизнес-логика, алгоритмы, DB-методы) | НЕ принимает архитектурных решений, НЕ правит файлы других агентов |

## Конвейер

```
Фаза 0          Фаза 1          Фаза 2          Фаза 3                 Фаза 4          Фаза 5
АНАЛИЗ    →     ПЛАН     →     ФУНДАМЕНТ  →    РЕАЛИЗАЦИЯ        →    ВЕРИФИКАЦИЯ →   ДОСТАВКА
Opus            Opus            Opus            Opus + Sonnet(ы)       Opus            Opus
```

### Фаза 0: Анализ (Opus)

**Вход:** задача от пользователя
**Выход:** граф зависимостей

Opus делает:
1. Читает `MODULE_MAP.md`
2. Определяет затронутые файлы и группы модулей
3. Строит граф зависимостей между изменениями
4. Определяет: какие файлы зависимы (Opus), какие изолированы (агенты)
5. **GPU-check:** если задача затрагивает `gpu/`, `pipeline/transcriber.py` → помечает как Opus-only

### Фаза 1: План (Opus)

**Вход:** граф зависимостей
**Выход:** план, одобренный пользователем

Opus использует Plan Mode:
1. Исследует код затронутых файлов
2. Создаёт план с разбивкой на приоритеты (P0, P1, P2...)
3. Для каждой задачи указывает: **opus-direct** или **agent-parallel**
4. Указывает порядок выполнения (что от чего зависит)
5. Пользователь одобряет через ExitPlanMode

**Критерий выхода:** план одобрен.

### Фаза 2: Фундамент (Opus)

**Вход:** одобренный план
**Выход:** стабильные интерфейсы

Opus пишет напрямую:
- Типы и dataclass-ы (если новые)
- `__init__.py` с реэкспортами
- Миграции БД (`database/migrations.py`)
- Баг-фиксы (P0)
- Всё, от чего зависят агенты
- `constants.py` — новые enum-значения

**API ЗАМОРАЖИВАЕТСЯ.** После этой фазы сигнатуры функций, которые будут реализовывать агенты, не меняются.

**Критерий выхода:** интерфейсы стабильны, агенты могут работать.

### Фаза 3: Реализация (Opus + Sonnet параллельно)

**Вход:** замороженные интерфейсы
**Выход:** рабочий код

> Общее правило параллельности — см. [CLAUDE.md § Параллельность](CLAUDE.md#параллельность). Здесь — его применение к фазе реализации.

Opus и Sonnet-агенты работают **одновременно**:

```
┌─ Opus:     __main__.py, config.py, glue code, GPU-модули
├─ Sonnet A: изолированный модуль X (например publisher.py)
└─ Sonnet B: изолированный модуль Y (например database/models.py)
```

Агенты запускаются через Agent tool:
```
Agent(subagent_type="general-purpose", model="sonnet", prompt="<структурированная спека>")
```

**Критерий выхода:** все задачи выполнены.

### Фаза 4: Верификация (Opus)

**Вход:** код от всех авторов
**Выход:** зелёные тесты

Многослойная проверка:

| Слой | Проверка | Время |
|------|----------|-------|
| 1 | AST syntax: `python3 -c "import ast; ast.parse(open(f).read())"` | мгновенно |
| 2 | Import resolution: `python3 -c "import slicr.module"` | 2 сек |
| 3 | Lint: `ruff check src/slicr/path/to/file.py` | мгновенно |
| 4 | pytest: `python -m pytest tests/ -x -q --timeout=120` | 30-60 сек |
| 5 | ffprobe: проверка выходных файлов (если менялся editor/publisher) | 5 сек |

Слои 1-3 — сразу после каждого агента.
Слой 4 — после интеграции всех результатов.
Слой 5 — опционально, при изменениях в video pipeline.

**Критерий выхода:** все проверки пройдены.

### Фаза 5: Доставка (Opus)

1. Обновить `MODULE_MAP.md` (если структурные изменения)
2. Обновить другие доки по правилу целостности документации (см. CLAUDE.md)
3. Коммит **по запросу пользователя** (не автоматически)
4. Push по запросу пользователя

## Решающая таблица: Opus или агент?

| Критерий | Opus напрямую | Sonnet-агент |
|----------|--------------|--------------|
| **Зависимости** | Файл импортирует/импортируется другими изменениями | Никто из текущей задачи не зависит |
| **Объём** | Любой | > 50 строк (иначе накладные > пользы) |
| **Тип кода** | Glue, `__main__.py`, `__init__.py`, config, orchestrator | Бизнес-логика, алгоритмы, DB-методы |
| **GPU** | `gpu/*`, `pipeline/transcriber.py` — ВСЕГДА Opus | Остальные pipeline-модули |
| **Безопасность** | Auth, валидация API ключей, session management | Не security-critical |
| **Контекст** | Нужно знать 3+ файлов | Достаточно 1-2 файла + интерфейс |
| **Файлов** | 1-3 связанных | 1 изолированный модуль |

**Правило большого пальца:** если задача затрагивает < 3 файлов и все зависимы — Opus напрямую, без агентов.

## GPU Safety Rules

**GPU-код — ТОЛЬКО Opus напрямую, НИКОГДА агентам:**

| Файл | Почему Opus-only |
|------|-----------------|
| `gpu/guard.py` | Pre-flight check, решение ALLOW/DENY — критичен для безопасности оператора |
| `gpu/monitor.py` | Runtime watchdog, решение ABORT — критичен для VRAM |
| `pipeline/transcriber.py` | Загрузка/выгрузка модели whisper, VRAM management |
| `pipeline/orchestrator.py` | GPU-очередь, координация с guard |

**Причина:** агент не видит полный контекст GPU-стека. Ошибка в GPU-коде = OOM-crash для оператора-графика.

## ffmpeg-верификация

При изменениях в `pipeline/editor.py`, `utils/video.py`, `utils/subtitles.py` — обязательная проверка выходного файла:

```bash
# Проверка длительности, кодека, разрешения
ffprobe -v quiet -print_format json -show_format -show_streams output.mp4

# Ожидаемые значения:
# - codec_name: h264
# - width: 1080, height: 1920 (или 9:16 ratio)
# - duration: 15-60 сек
```

## Промпт-задачи (Claude API)

Изменения промптов для `services/claude_client.py` и `pipeline/selector.py` — это **отдельный тип задачи**:

1. Промпт — это не код, а инструкция для AI
2. Тестирование промпта — ручное (на реальных транскрипциях)
3. Не делегировать агентам — нужен контекст всего pipeline

**Протокол:**
1. Opus вносит изменения в промпт
2. Запускает selector в mock-режиме с тестовой транскрипцией
3. Оценивает качество выбора фрагмента
4. При необходимости — итерирует

## Промпт для воркера: структура

Агент получает **структурированный пакет**, а не свободную спеку:

```
Ты Sonnet-воркер проекта Slicr (видео-нарезчик).

ЗАДАЧА: <что реализовать — конкретно>

ФАЙЛ: <какой файл править — ОДИН>

ЗАМОРОЖЕННЫЙ ИНТЕРФЕЙС:
<точные сигнатуры функций, которые нужно реализовать>
<типы входов и выходов>

КОНТЕКСТ:
<существующий код файла, если дополнение>
<__init__.py пакета>

ПАТТЕРНЫ ПРОЕКТА:
- async/await для всех I/O
- logger = logging.getLogger(__name__)
- Параметризованные SQL (без f-string)
- Абсолютные импорты: from slicr.xxx import ...
- Type hints + StrEnum для всех функций
- Python 3.13+

НЕ ДЕЛАТЬ:
- НЕ править другие файлы
- НЕ менять сигнатуры замороженных функций
- НЕ добавлять то, чего нет в задаче
- НЕ коммитить
- НЕ трогать GPU-модули (gpu/, pipeline/transcriber.py)
```

## Антипаттерны

| Антипаттерн | Почему плохо | Что делать |
|---|---|---|
| Два агента правят один файл | Конфликты | Один файл — один автор |
| Агент правит файл, зависящий от другого агента | Race condition на интерфейсах | Зависимые файлы → Opus или один агент |
| Агент без замороженного интерфейса | Результат не совместим | Фаза 2 обязательна перед агентами |
| Пропуск AST check после агента | Синтаксические ошибки дойдут до pytest | Слой 1-3 сразу после каждого агента |
| Агент на < 50 строк | Накладные расходы > пользы | Opus напрямую |
| Коммит без pytest | Сломанный main | pytest обязателен |
| Делегирование GPU-кода агенту | Агент не видит полный GPU-контекст | Opus напрямую (см. GPU Safety Rules) |
| Агент меняет промпт для Claude API | Нужен контекст всего pipeline | Opus напрямую (см. Промпт-задачи) |
